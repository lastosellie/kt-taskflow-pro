## 절대 규칙 (위반 불가)

### 1. 추측 금지
불확실한 사항은 절대 추측하지 않는다. 모르면 반드시 묻는다.
요구사항이 모호하면 **선택지 2~3개**를 제시하고 결정을 요청한다.

### 2. 돌발 의존성 금지
새 라이브러리·패키지·서비스를 추가할 때 사전 승인 없이 `package.json` 또는 설정 파일을 수정하지 않는다.
추가가 필요하다고 판단되면 이유와 대안을 함께 제시하고 확인을 받은 후 진행한다.

### 3. 테스트 없이 완료 금지
새 기능·버그 수정이 포함된 작업은 해당 테스트(단위 또는 통합)가 작성되고 통과해야 완료로 간주한다.
테스트 작성이 불가능한 경우 그 이유를 명시한다.

### 4. 시크릿 하드코딩 금지
API 키, 비밀번호, 토큰, DB 연결 문자열 등 모든 시크릿은 환경 변수(`.env`)로만 관리한다.
코드·커밋·로그 어디에도 시크릿 값이 포함되어서는 안 된다.

### 5. 폴더 구조 임의 변경 금지
`docs/06-conventions.md`에 정의된 폴더 구조를 임의로 변경하지 않는다.
구조 변경이 필요하다고 판단되면 변경 이유와 영향 범위를 먼저 설명하고 승인을 받는다.

## 모호한 요청 처리 방식

요청이 불명확하거나 해석이 2가지 이상 가능한 경우:

1. **추측하지 않는다.**
2. 가능한 해석을 **2~3개의 선택지**로 정리해 제시한다.
3. 각 선택지의 트레이드오프를 한 줄씩 설명한다.
4. 사용자의 선택을 받은 후에만 작업을 시작한다.

**예시 형식**:
```
요청이 다음 중 어느 방향인지 확인이 필요합니다:

A) ... → (장점/단점)
B) ... → (장점/단점)
C) ... → (장점/단점)

어떤 방향으로 진행할까요?
```
