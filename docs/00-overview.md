## 읽는 순서

```
00-overview  →  01-product  →  02-specs  →  03-design  →  04-tasks  →  05-conventions
   (지도)         (무엇을)       (어떻게)      (어떻게 보이나)   (순서)        (규칙)
```

모든 작업 시작 전 이 순서대로 읽는다. 건너뛰지 않는다.

---

## 왜 이렇게 나누는가 — 관심사 분리(Separation of Concerns) 원칙

각 문서는 **하나의 질문에만 답한다.**

| 문서 | 답하는 질문 | 섞으면 안 되는 이유 |
|------|------------|-------------------|
| `01-product` | *무엇을* 만드는가 | 비즈니스 결정과 기술 결정을 분리해야 제품 방향이 바뀌어도 기술 명세를 통째로 수정할 필요가 없다 |
| `02-specs` | *어떻게* 구현하는가 | 기술 스택이 바뀌어도 제품 요구사항 문서는 그대로 유지된다 |
| `03-design` | *어떻게 보이는가* | UI 변경이 API 명세에 영향을 주지 않도록 시각적 관심사를 격리한다 |
| `04-tasks` | *어떤 순서로* 만드는가 | 일정·우선순위는 자주 바뀌므로 다른 문서와 분리해 변경 비용을 낮춘다 |
| `05-conventions` | *어떤 규칙으로* 협업하는가 | 컨벤션은 코드와 무관하게 팀 전체에 적용되므로 별도 관리한다 |

**핵심**: 한 문서를 수정할 때 다른 문서를 함께 수정해야 한다면, 관심사가 잘못 분리된 것이다.
